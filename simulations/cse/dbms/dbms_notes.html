1.1 What is Data, Database and DBMS? 🔥 High-yield

Before you can define a DBMS, you need three building-block terms nailed down — examiners love asking "define" questions on exactly these.

📌 Key Definitions
  • Data: known facts that can be recorded and have an implicit meaning; raw, un-processed values.
  • Database: a highly organized, interrelated, and structured set of data about a particular enterprise, controlled by a DBMS.
  • DBMS: a software application/system that facilitates the creation, organization and management of databases — a collection of interrelated data and a set of programs to access that data.

What a DBMS actually does

  • Serves as an intermediary between users and the database, ensuring efficient storage, retrieval and manipulation of data.
  • Examples: Oracle, MySQL, SQL Server, PostgreSQL.

When do you need a database system?

Database systems are used to manage collections of data that are:

  • Highly valuable — e.g. business data
  • Relatively large — e.g. a university's records
  • Accessed by multiple users, often simultaneously — e.g. online platforms
  • Ones where relationships and queries matter — e.g. hospital records
  • Ones that need security and backup — e.g. banking data
💡 Example — Database Applications
  • Enterprise: Sales (customers, products), Accounting (payments, assets), HR (employees, salaries)
  • Banking & Finance: accounts, loans, credit-card transactions, real-time market data
  • Universities: registration, grades
  • Recent applications: Multimedia DBs (social media, search engines), GIS, genome databases, data warehouses, mobile DBs (SQLite), real-time/active databases (stock markets)
1.2 Characteristics of a Database System 🔥 High-yield

A very common 5-mark question: "List and explain the characteristics of a database system."

  1. Data Abstraction — a data model hides storage details and presents users with a conceptual view of the database.
  2. Multiple Views of Data — each user may see a different view describing only the data of interest to them.
  3. Sharing of data & multi-user transaction processing — allows concurrent users to retrieve and update the database at the same time.
  4. Insulation between programs and data — data and application programs are independent of each other (data independence).
  5. Self-describing nature — a DBMS contains not just the data but also metadata describing the structure of the database.
CharacteristicWhat it meansExample
Data AbstractionHides storage/implementation details from the userA user queries "students in CSE" without knowing how rows are stored on disk
Multiple ViewsDifferent users see different, tailored views of the same DBA registrar sees fee details; a professor sees only grades
Sharing & Multi-userMany users/transactions run concurrently on the same dataTwo clerks booking the last seat on a flight at the same time
Insulation (Data Independence)Programs and data structures can change independentlyAdding a column to EMPLOYEE doesn't break existing reports
Self-describing NatureDBMS stores metadata (schema) along with actual dataThe data dictionary/catalog describing table structures
💡 Example — Why this matters

In a file-processing system (no DBMS), each application defines and manages its own data files — leading to data redundancy and inconsistency. A DBMS centralizes this so all the above characteristics hold true for every application sharing the database.

📝 Note

Remember this list with the mnemonic "A-V-S-I-S": Abstraction, Views, Sharing, Insulation, Self-describing.

1.4 Structured, Semi-Structured & Unstructured Data 🔥 High-yield
TypeDefinitionExample
Structured DataRepresented in a strict, predefined format; a DBMS checks it against the schemaRelational tables (e.g. an Employee table)
Semi-Structured DataHas some structure, but not every record has identical structure; schema info is mixed in with the data itself ("self-describing")JSON, XML documents

⚠️ Common Mistake

Students often confuse semi-structured with unstructured. Semi-structured data does have some organization (tags/keys in XML/JSON) — it's just not uniform across records, unlike unstructured data which has almost no discernible structure.

1.5 Data Models — High-level, Low-level & Representational 🔥 High-yield
📌 Definition

A data model is a collection of concepts used to describe the structure of a database — its data types, relationships, and constraints — plus a set of basic operations for retrievals and updates.

CategoryDescriptionUsed By
High-level / ConceptualClose to how users perceive data; uses entities, attributes, relationshipsDesigners (e.g. ER Model)
Low-level / PhysicalDescribes how data is stored on storage media — file organization, indexingDBAs / specialized users
RepresentationalEasily understood by end users; hides storage details; often record-basedEnd users & developers

Representational models are further classified into:

Hierarchical Model
  • Oldest data model — a tree structure ("database tree")
  • Collection of records connected by links
  • A collection of rooted trees is a forest
  • Rectangular box = record type, line = link between record types
Network Model
  • An evolved version of hierarchical — an item can have multiple parents
  • Disadvantage: needs lots of storage, inflexible / hard to modify
Relational Model
  • Consists of a collection of tables (relations)
  • Information stored in tabular form
  • Most popular model today — simple, flexible, backed by strong mathematical theory (relational algebra)
Hierarchical (tree/parent-child) vs Network (multiple parents) vs Relational (tables)
1.6 Schema vs Instance 🔥 High-yield
📌 Key Definitions
  • Schema: the overall design/description of the database — specified during design, and not expected to change frequently. Displayed as a schema diagram.
  • Instance: the actual content/data stored in the database at one particular moment in time.

Schema is analogous to a type in a programming language; an instance is analogous to the value of a variable — the schema rarely changes, but the instance changes every time data is inserted/updated/deleted.

Logical vs Physical Schema

TypeDescribes
Logical SchemaStructure in terms of tables, fields, data types, relationships, constraints — independent of physical storage
Physical SchemaHow data is stored on physical devices — file organization, compression, indexing
1.7 Levels of Data Abstraction 🔥 High-yield

A major purpose of a database system is to give users an abstract view of data — hiding how it's actually stored. There are three levels:

Closer to the user (top) → closer to hardware (bottom)
LevelFocus
PhysicalHow a record is stored — data structures, file formats, indexing methods, access mechanisms
LogicalWhat data is stored and the relationships among data (e.g. type instructor = record ID:string; name:string; end;) — ER diagrams & normalization live here
ViewHighest level, closest to end users; application programs hide data-type details; can hide sensitive info (e.g. salary) for security
💡 Example

Clerks in a university registrar's office see only the part of the database dealing with students (their view) — they cannot access instructor salary information, even though at the physical level all this data sits in the same storage system.

1.8 Three-Schema Architecture 🔥🔥 Very Frequently Asked
📌 Goal

The goal of the three-schema architecture is to separate the user applications from the physical database, by defining DBMS schemas at three levels.

External (user views) → Conceptual (whole DB structure) → Internal (physical storage)
Schema LevelDescribes
Internal SchemaPhysical storage structures & access paths — typically a physical data model
Conceptual SchemaStructure and constraints for the whole database for a community of users — a conceptual/implementation data model
External SchemasVarious user views — usually same data model as conceptual schema

Mappings between levels transform requests and data: programs refer to an external schema and are mapped by the DBMS to the internal schema for execution; data from the internal level is reformatted to match a user's external view.

1.9 Data Independence 🔥 High-yield
📌 Definition

Data independence is the capacity to change the schema at one level of a database system without having to change the schema at the next higher level.

Logical Data Independence
  • Change the conceptual schema without changing external schemas or application programs
Physical Data Independence
  • Change the internal schema without changing the conceptual schema
  • E.g. reorganizing files or adding new indexes to improve performance
⚠️ Common Mistake

Don't mix up the direction: logical independence shields applications from conceptual-schema changes; physical independence shields the conceptual schema from storage-level changes. Physical independence is generally easier to achieve than logical.

💡 Example

Physical: the DBA switches EMPLOYEE's storage from a heap file to a B+-tree index for faster lookups — no application code or conceptual schema changes. Logical: a new attribute Email is added to the conceptual EMPLOYEE schema — existing external views/programs that don't reference Email keep working unchanged.

1.10 Database Languages: DDL, DML, DQL, DCL, TCL 🔥🔥 Very Frequently Asked
LanguageFull FormPurposeSample Commands
DDLData Definition LanguageDefines/modifies the structure or schema of the databaseCREATE, ALTER, DROP
DMLData Manipulation LanguageManages & manipulates data — retrievals and updatesINSERT, UPDATE, DELETE
DQLData Query LanguageRetrieves data from the databaseSELECT
DCLData Control LanguageControls access rights & permissionsGRANT, REVOKE
TCLTransaction Control LanguageManages transactions in the databaseCOMMIT, ROLLBACK, SAVEPOINT

DDL is used by database designers to specify the conceptual schema — and in many DBMSs also the internal/external schemas. Some systems use a separate Storage Definition Language (SDL) and View Definition Language (VDL) for that.

DML commands can be embedded in a general-purpose language (COBOL, C, C++, Java) via a library of functions (e.g. JDBC's getConnection(), createStatement(), executeQuery()), or used stand-alone as a query language.

DBMS Programming Language Approaches

  • Embedded Approach: e.g. embedded SQL (for C, C++), SQLJ (for Java)
  • Procedure Call Approach: e.g. JDBC (Java), ODBC (Open Database Connectivity) as APIs
  • Database Programming Language Approach: e.g. Oracle's PL/SQL — incorporates SQL and its data types as integral components
  • Scripting Languages: PHP (client-side), Python (server-side) used to write database programs

1.12 ER Model & ER Diagram — Basic Concepts 🔥🔥 Very Frequently Asked
📌 Definitions
  • ER Model: used mainly for communication between database designers and end users during the analysis phase of database development. Composed of entity types and the relationships that can exist between entity instances. It is independent of the DBMS/data model used to actually implement the database.
  • ER Diagram: a conceptual model giving the graphical representation of the logical structure of a database — showing all constraints and relationships among components.

Basic building-block terms

TermMeaningExample
EnterpriseAny kind of organizationColleges, banks, companies
EntityAn "object" or "thing" in the real world, distinguishable from othersA student, a loan, an employee, a car
AttributeA characteristic/property that describes an entityStudent → name, age, address; Car → color, model
ValueThe actual data/information stored in an attribute"Alice", 21, "Blue"
Entity SetA collection of entities of the same typeAll students = the Student entity set
DomainThe set of all possible values for an attributeDomain of Age = positive integers < 120
1.13 Types of Attributes 🔥🔥 Very Frequently Asked

Six important classifications you must be able to identify from an ER diagram or a word problem:

TypeDefinitionExample
SimpleCannot be divided furtherAge, Gender
CompositeCan be divided into smaller sub-partsName → First_Name + Last_Name; Address → Street + City + Pincode
Single-valuedHas only ONE value for a given entityAadhar Number
Multi-valuedCan have MULTIPLE values for one entity (shown with a double ellipse)Phone_Number(s), Email_ID(s)

NULL means "nothing" — an attribute has a null value when the value is either unknown or not applicable to that entity.

Ellipse notations: single (plain) · multi-valued (double) · derived (dashed) · composite (has sub-attributes)

1.15 Mapping Cardinalities (Cardinality Ratio) 🔥🔥 Very Frequently Asked
📌 Definition

Mapping cardinality specifies the number of entities of one entity set that can be associated with entities of another entity set via a relationship set. It's most helpful for describing binary relationship sets.

CardinalityMeaningExample
One to One (1:1)An entity in X is associated with at most one entity in Y, and vice versaA country has only one president; a person is president of at most one country
One to Many (1:N)An entity in X can relate to many entities in Y, but an entity in Y relates to at most one in XA manager has many employees, but each employee has only one manager
Many to One (N:1)Many entities in X relate to at most one entity in YMany employees can be assigned to a single project
Many to Many (M:N)Entities on both sides can relate to any number of entities on the other sideA student can take many subjects; a subject can have many students
The four cardinality ratios — always identify these correctly for any relation, they're a guaranteed exam question
⚠️ Common Mistake

Cardinality is written from the perspective of "how many of entity B for one A" — read carefully which side the "1" or "N" sits on; swapping the direction is one of the most common exam errors.

1.16 Participation Constraints (Total vs Partial) 🔥🔥 Very Frequently Asked
📌 Definition

Participation constraint describes whether the existence of an entity depends on it being related to another entity via a relationship.

Total Participation
  • Every entity in the set is involved in the relationship
  • Represented by a double line
  • Eg: every Loan is concerned with at least one Customer → Loan entity set is totally participating
Partial Participation
  • Not all entities are involved in the relationship
  • Represented by a single line
  • Eg: a Customer may or may not take a Loan → Customer entity set is partially participating
Single line (Customer↔Borrows) = partial participation · Double line (Borrows↔Loan) = total participation
💡 Example

All students are enrolled in a Core Course (total participation), but only some students take an Elective Course (partial participation).

1.17 Can Relationships Have Attributes?

You assign an attribute to a relationship (not an entity) when the relationship itself carries information that isn't naturally an attribute of either participating entity.

💡 Example

Entities: Student(StudentID, Name), Course(CourseID, Title). Relationship: Enrolled. The relationship attributes are EnrollmentDate and Grade — they only make sense in the context of a specific student being enrolled in a specific course, not as a property of Student or Course alone.

1.18 Keys — Super, Candidate, Primary, Alternate, Secondary, Composite, Foreign 🔥🔥 Very Frequently Asked
📌 Definition

A key is an attribute or a set of attributes used to uniquely identify data (tuples) in an entity set. Attributes used as keys are key attributes; the rest are non-key attributes.

Key TypeDefinition
Super KeyAny set of one or more attributes that, taken together, can uniquely identify a tuple. Not necessarily minimal — can contain extra attributes.
Candidate KeyA minimal super key — no proper subset of it is also a super key. Every candidate key is a super key, but not vice versa.
Primary KeyThe candidate key chosen by the designer to uniquely identify tuples. Cannot be NULL, has no duplicate values, and is usually underlined.
Alternate KeyAll candidate keys other than the chosen primary key.
Secondary KeyAn attribute (or set) that doesn't uniquely identify data but identifies a group of data (used in searching, e.g. Department No.)
Composite KeyA key made up of two or more attributes together (no single attribute is sufficient alone).
Foreign KeyAn attribute in one relation that is the primary key of another relation — used to link the two tables.
💡 Worked Example

Relation: Students(Student_ID, Roll_No, Email, Name, Department)

  • Super Keys: {Student_ID}, {Roll_No}, {Email}, {Student_ID, Name}, {Roll_No, Department} — any combination that stays unique
  • Candidate Keys: {Student_ID}, {Roll_No}, {Email} — each is minimal & unique on its own
  • Primary Key: {Student_ID} — chosen by the designer
📝 Exercise — Students(SSN, Name, Home_Address, Birthdate, GPA)

(a) Candidate keys: {SSN}, {Name, Home_Address, Birthdate}
(b) A super key that is NOT a candidate key: {SSN, Name} (has a redundant attribute)

⚠️ Common Mistake

"Any key is a super key, but not every super key is a (candidate) key." Students often write these as synonyms — remember: super key ⊇ candidate key ⊇ primary key (primary key is exactly one chosen candidate key).

1.19 Weak Entity Sets vs Strong Entity Sets 🔥🔥 Very Frequently Asked
TypeDefinition
Strong Entity SetHas sufficient attributes to form its own primary key — can exist independently
Weak Entity SetDoes NOT have sufficient attributes to form a primary key on its own — its existence depends on an "identifying" (owner) strong entity
  • Instead of a primary key, a weak entity uses the identifying entity's key + its own discriminator (partial key) to uniquely identify itself.
  • In ER diagrams: weak entity = double-outlined rectangle; identifying relationship = double-outlined diamond; discriminator = dashed underline.
Payment is a weak entity — it can't exist without a Customer (double rectangle + double diamond)

1.22 Steps to Draw an ER Diagram 🔥 High-yield

A common design-question format in the exam: "Design an ER diagram for [scenario]." Follow these 7 steps every time.

  1. Understand the problem statement — read the scenario carefully, identify what needs to be stored and how things relate.
  2. Identify Entities (usually nouns) — e.g. Student, Course, Book. Represent using rectangles.
  3. Identify Relationships (usually verbs/actions) — e.g. Enrolls, Teaches, Borrows. Represent using diamonds.
  4. Identify Attributes — ask "what details do I want to store?" Represent using ellipses; underline the primary key; use double ellipses for multi-valued and dashed ellipses for derived attributes.
  5. Identify Keys — decide the primary key for each entity; check for candidate/alternate/composite keys.
  6. Determine Cardinality & Participation — define 1:1, 1:N or M:N and mark total/partial participation with solid/double lines.
  7. Draw the diagram — connect entities via relationships, attach attributes, keep it organized and use correct symbols.
💡 Worked Example — Bank Loan Scenario

"A bank offers loans; customers borrow money and repay in installments."

  • Customer — Customer_id (PK), Name, Address
  • Loan — Loan_no (PK), Type, Amount
  • Payment — Payment_no (PK, weak entity), Amount, Date
  • Relationships: Customer borrows Loan (M:N or 1:N depending on scenario); Loan has Payment (Payment is a weak entity dependent on Loan)
1.23 Complete Worked ER Diagram — Student Enrolls in Course 🔥🔥 Very Frequently Asked

Putting every notation symbol together into one full diagram is exactly what a "Draw the ER diagram for..." exam question expects. Scenario: "A student enrolls in courses. Each student has a roll number, a name split into first/last, multiple phone numbers, and an age derived from date of birth. Each course has a course code and title. A student may enroll in many courses and a course may have many students; we also record the enrollment date and grade for each enrollment."

Full ER diagram: STUDENT (composite Name, multivalued Phone, derived Age) —M:N— Enrolls —N:M— COURSE, with relationship attributes Enroll_Date & Grade
📝 How to read this diagram
  • Roll_No is underlined → it's the primary key of STUDENT.
  • Name is composite → splits into First and Last.
  • Phone has a double ellipse → multi-valued (a student can have several numbers).
  • Age has a dashed ellipse → derived (computed from Date_of_Birth, not stored directly).
  • Enrolls is M:N → any student can take many courses and any course can have many students, so it needs its own attributes (Enroll_Date, Grade) since these only make sense per student-course pair.

2.1 Structure of Relational Databases — Core Concepts 🔥🔥 Very Frequently Asked

The relational model represents how data is stored in relational databases — as relations (tables).

TermMeaning
RelationA table; a set of rows (tuples)
TupleA row — an ordered set of values enclosed in angle brackets, e.g. <632895, "John Smith", "101 Main St", "(404) 894-2000">
AttributeA column header — indicates the meaning of the data items in that column
DomainThe set of valid/permissible values for an attribute, plus its data type/format

Formal Schema Notation

R(A1, A2, ..., An)

R = relation name; A1...An = attributes of the relation.

💡 Example

CUSTOMER(Cust-id, Cust-name, Address, Phone#) — CUSTOMER is the relation name defined over 4 attributes. Domain of Cust-id, for example, could be "6-digit numbers".

2.2 Relational Integrity Constraints 🔥🔥 Very Frequently Asked
📌 Definition

Constraints are conditions that must hold on all valid relation states. There are three main types, plus a schema-based domain constraint.

ConstraintRule
Domain ConstraintEvery value in a tuple must come from the domain of its attribute (or be NULL, if allowed)
Key ConstraintNo two tuples in a relation state can have the same value for the (candidate/primary) key
Entity IntegrityPrimary key attributes of a relation cannot be NULL in any tuple — used to uniquely identify each row
Referential IntegrityA foreign key value must either match an existing primary key value in the referenced relation, or be NULL

Referential Integrity — the detail

  • Involves two relations: a referencing relation R1 and a referenced relation R2.
  • R1 has a foreign key (FK) that references the primary key (PK) of R2.
  • Tuple t1 in R1 references tuple t2 in R2 if t1[FK] = t2[PK].
  • The FK value must be either (1) an existing PK value in R2, or (2) NULL — and if NULL, the FK must not be part of R1's own primary key.
⚠️ Common Mistake

Entity integrity applies to the primary key of the same relation (it can never be null). Referential integrity is about a foreign key pointing to another relation's primary key. Don't mix these two up in definition questions.

💡 Worked Example — Spot the Violation

Schema: EMPLOYEE(SSN, Name, DNO), DEPARTMENT(DNUMBER, DNAME) where EMPLOYEE.DNO is a foreign key referencing DEPARTMENT.DNUMBER.

Attempted RowConstraint Violated?Why
EMPLOYEE(NULL, "Ravi", 5)✅ Entity IntegritySSN is the primary key — cannot be NULL
EMPLOYEE(101, "Asha", 9)✅ Referential IntegrityDNO = 9 doesn't exist in DEPARTMENT
EMPLOYEE(101, "Asha", NULL)❌ No violationFK can be NULL if it's not part of EMPLOYEE's own primary key (means "not yet assigned a dept")
Two rows both with SSN = 101✅ Key ConstraintPrimary key values must be unique
2.3 Key Constraints — Super Key & Candidate Key (Relational View) 🔥 High-yield
💡 Worked Example — CAR relation

CAR(State, Reg#, SerialNo, Make, Model, Year)

  • CAR has two keys: Key1 = {State, Reg#}, Key2 = {SerialNo} — both are also super keys
  • {SerialNo, Make} is a super key, but not a key (it's not minimal)
  • The designer chooses SerialNo as the Primary Key; its value is used to uniquely identify each tuple
📝 Rules Recap
  • Any key is a super key (but not vice versa)
  • Any set of attributes that includes a key is a super key
  • A minimal super key is a candidate key
2.4 Relevance of Synthesizing ER Diagram → Relational Schema

Why bother converting a conceptual ER diagram into tables? A frequent "explain the importance" theory question.

  • Bridges the gap — ER is conceptual (high-level); relational schema is logical (implementable in SQL)
  • Ensures data integrity — identifying keys/foreign keys/constraints during conversion enforces entity & referential integrity (e.g. weak entities can't exist without their strong entity)
  • Removes ambiguity — ER diagrams look clear visually but lack formal storage rules; the relational schema gives explicit tables, attributes, constraints
  • Supports query processing — once converted, data can be queried with Relational Algebra or SQL
  • Enables normalization & optimization — the relational schema lets you check redundancy and apply normal forms
2.5 Synthesizing ER Diagrams into Tables — Basic Reductions 🔥 High-yield

1. Strong Entity Sets

For a strong entity set E with attributes a1...an, create a table with the same name as E, having n columns (one per attribute).

2. Composite Attributes

Create a separate column for each component/part of the composite attribute (e.g. Name → First_Name, Last_Name).

3. Multi-valued Attributes

  • Cannot be stored as a single column — create a separate table.
  • New table has: (a) the primary key of the owning entity/relationship as a foreign key, and (b) the multi-valued attribute itself.
  • Rows = total number of values of that attribute across all entities.

4. Weak Entity Sets

  • Becomes a table that includes a column for the PK of the identifying (owner) strong entity.
  • The weak entity's own primary key = {owner's PK + weak entity's discriminator}.

5. Generalization

For each lower-level entity set, create a table with a column for each of its own attributes, plus a column for the PK of the higher-level entity set.

💡 Example

Full_Time_Employee(Employee_ID, Qualification) and Part_Time_Employee(Employee_ID, Hours_Worked) — both share Employee_ID from the higher-level Employee entity.

2.6 Reduction of Relationships & Choosing the Primary Key 🔥 High-yield

Let R be a relationship set involving entity sets E1...EN. Create a table named R, with columns = the primary keys of all participating entity sets + any descriptive attributes of R.

Relation schema for R = {a1, a2, ..., am} ∪ {b1, b2, ..., bn}
where a1..am = union of PKs of participating entities
      b1..bn = descriptive (own) attributes of R

Choosing the primary key of the relationship's table — depends on cardinality

CardinalityPrimary Key RuleExample
Many-to-ManyUnion of PKs of both participating entity setsTAKE(sid, cid, year, grade) — PK = (sid, cid, year)
One-to-ManyPK of the entity on the "many" sideTEACH(pid, cid, year) — PK = (cid, year)
Many-to-OnePK of the entity on the "many" sideSame rule as above, mirrored
One-to-OnePK of either entity set can be chosenMARRY(mid, wid, year) — mid or wid can be PK
2.7 ER-to-Relational Mapping Algorithm (7 Steps) 🔥🔥 Very Frequently Asked — guaranteed long-answer question

This is the full formal algorithm (Elmasri & Navathe) for converting an entire ER schema into relations. Learn the 7 steps in order — this is a classic 10-mark question using the COMPANY database example (Employee, Department, Project, Dependent, Works_On, Works_For, Manages, Controls, Supervision).

Step 1 — Mapping of Regular (Strong) Entity Types

  • For each strong entity type E, create a relation R with all the simple attributes of E.
  • Choose one key attribute of E as the primary key of R (if composite, all its component attributes together form the PK).
  • Example: create EMPLOYEE, DEPARTMENT, PROJECT relations from the COMPANY ER diagram.

Step 2 — Mapping of Weak Entity Types

  • For each weak entity type W with owner E, create relation R with W's simple attributes.
  • Include the owner's primary key as a foreign key in R.
  • PK of R = {owner's PK} ∪ {W's partial key/discriminator}.
  • Example: DEPENDENT(ESSN, DEPENDENT_NAME, ...) where ESSN is a FK referencing EMPLOYEE.SSN, and PK = {ESSN, DEPENDENT_NAME}.

Step 3 — Mapping of Binary 1:1 Relationship Types

Three possible approaches for a 1:1 relationship R between entities S and T:

  1. Foreign Key approach: add the PK of T as a FK in S — preferably choose S to be the entity with total participation. E.g. MANAGES mapped by adding a FK in DEPARTMENT (since its participation is total).
  2. Merged relation option: merge both entity types + the relationship into one single relation (good when both sides have total participation).
  3. Cross-reference/relationship relation: create a third relation purely to cross-reference the PKs of S and T.

Step 4 — Mapping of Binary 1:N Relationship Types

  • Identify relation S on the "N" (many) side.
  • Include the PK of the "1" side's relation T as a FK in S, plus any relationship attributes.
  • Example: for WORKS_FOR, include DEPARTMENT's PK (DNUMBER) as FK DNO in the EMPLOYEE relation.

Step 5 — Mapping of Binary M:N Relationship Types

  • Create a new relation S to represent the relationship R.
  • Include the PKs of both participating entities as FKs in S — their combination becomes S's PK.
  • Include any descriptive attributes of R as attributes of S.
  • Example: WORKS_ON(ESSN, PNO, HOURS) — PK = {ESSN, PNO}, both being FKs to EMPLOYEE and PROJECT respectively.

Step 6 — Mapping of Multivalued Attributes

  • For each multivalued attribute A, create a new relation R.
  • R includes an attribute for A, plus the owning entity's PK as a FK.
  • PK of R = {A, owner's PK}.
  • Example: DEPT_LOCATIONS(DNUMBER, DLOCATION) — PK = {DNUMBER, DLOCATION}.

Step 7 — Mapping of N-ary Relationship Types (n > 2)

  • Create a new relation S for the n-ary relationship type.
  • Include the PKs of all n participating entities as FKs in S, plus any of R's own attributes.
  • Example: SUPPLY relationship mapped to relation SUPPLY, PK = {SNAME, PARTNO, PROJNAME}.

⚠️ Common Mistake

Students often forget that for 1:1 relationships there are three valid mapping strategies (not just one) — examiners specifically test whether you know all three approaches.

2.8 What is Relational Algebra? Why Learn It? 🔥 High-yield
📌 Definition

Relational Algebra (RA) is a formal, mathematical query language used to manipulate and retrieve data stored in a relational database. It consists of operations (select, project, join, etc.) that take relations as input and produce a new relation as output.

  • The result of an operation is always a relation → this makes the algebra "closed" (every object in RA is itself a relation).
  • A sequence of RA operations forms a relational algebra expression, whose result represents the answer to a query.

Why Relational Algebra matters

  • Foundation of SQL: SQL is based on relational algebra (and relational calculus) — understanding RA explains how SQL is evaluated internally.
  • Mathematical precision: gives a precise, unambiguous way to express a query, unlike vague English descriptions.
  • Query optimization: DBMSs transform queries into more efficient RA forms (e.g. rearranging joins/projections).
  • Theoretical foundation: forms the basis of relational database theory proposed by E.F. Codd (1970).

Groups of RA Operations

GroupOperations
UnarySELECT (σ), PROJECT (π), RENAME (ρ)
Set TheoryUNION (∪), INTERSECTION (∩), DIFFERENCE (−), CARTESIAN PRODUCT (×)
BinaryJOIN (⋈), DIVISION (÷)
AdditionalOUTER JOINS, OUTER UNION, Aggregate Functions (SUM, COUNT, AVG, MIN, MAX)

Operators used inside conditions

  • Comparison: =, ≠, <, ≤, >, ≥
  • Connectives: AND (∧), OR (∨), NOT (¬)
2.9 SELECT Operation (σ) 🔥🔥 Very Frequently Asked
📌 Definition

SELECT (σ, sigma) is a unary operation that picks a subset of rows (tuples) from a relation, based on a selection condition acting as a filter.

σ <selection_condition> (R)
💡 Examples
σ DNO = 4 (EMPLOYEE)
σ SALARY > 30000 (EMPLOYEE)
σ Dept='CS' AND Salary>50000 AND Age<30 (EMPLOYEE)

SELECT Operation Properties

  • Produces a relation with the same schema (same attributes) as R.
  • Commutative: σ<c1>(σ<c2>(R)) = σ<c2>(σ<c1>(R))
  • A cascade of SELECTs can be applied in any order, or replaced by ONE selection with a conjunction of all conditions: 
    σc1(σc2(σc3(R))) = σ c1 AND c2 AND c3 (R)
  • Result tuple count ≤ tuple count of input relation R.
📝 Note

π and σ operating on a table both keep the same number of columns/rows relationship intuition: σ filters rows (height↓), π filters columns (width↓).

💡 Worked Table Example — σ DNO=4 (EMPLOYEE)

Input relation EMPLOYEE:

SSNFNAMELNAMESALARYDNO
101RaviKumar350004
102AshaMenon420005
103JohnSmith280004

Result of σ DNO=4(EMPLOYEE) — only rows where DNO = 4 survive, all 5 columns kept:

SSNFNAMELNAMESALARYDNO
101RaviKumar350004
103JohnSmith280004
2.10 PROJECT Operation (π) 🔥🔥 Very Frequently Asked
📌 Definition

PROJECT (π, pi) is a unary operation that keeps only certain columns (attributes) from a relation and discards the rest — a "vertical partitioning".

π <attribute_list> (R)
💡 Example
π LNAME, FNAME, SALARY (EMPLOYEE)

PROJECT Operation Properties

  • The project operation removes duplicate tuples — the result must be a mathematical set, and sets don't allow duplicates.
  • Number of tuples in result ≤ number of tuples in R (equal only if the attribute list includes a key of R).
  • PROJECT is NOT commutative: π<L1>(π<L2>(R)) ≠ π<L2>(π<L1>(R)) in general — however π<L1>(π<L2>(R)) = π<L1>(R) as long as L2 contains all attributes in L1.
⚠️ Common Mistake

This is one of the most common trap questions: "Is SELECT commutative? Is PROJECT commutative?" — SELECT is commutative, PROJECT is NOT. Memorize this pair distinctly.

💡 Worked Table Example — π DNO, DNAME(EMPLOYEE) shows duplicate removal

Input relation EMPLOYEE:

SSNFNAMEDNODNAME
101Ravi4Research
102Asha5Admin
103John4Research

Result of π DNO, DNAME(EMPLOYEE) — 3 input rows become only 2 output rows, because (4, Research) appeared twice and duplicates are removed:

DNODNAME
4Research
5Admin
2.11 RENAME Operation (ρ) & Combining Operations 🔥 High-yield

RENAME (ρ, rho) renames the relation, its attributes, or both — useful when a query needs multiple chained operations (and necessary for self-joins).

2.12 Set Theory Operations: UNION, INTERSECTION, DIFFERENCE 🔥🔥 Very Frequently Asked
⚠️ Required — Type / Union Compatibility

UNION, INTERSECTION and DIFFERENCE all need R1 and R2 to be "type compatible" (union compatible): (1) same number of attributes, and (2) domains of corresponding attributes must be compatible. The result keeps the attribute names of the first operand relation, by convention.

OperationSymbolResult
UNIONR ∪ SAll tuples in R, or S, or both — duplicates eliminated
INTERSECTIONR ∩ SOnly tuples that appear in BOTH R and S
SET DIFFERENCE (MINUS/EXCEPT)R − STuples in R but NOT in S
💡 Example — UNION

Retrieve SSNs of employees who work in dept 5 OR supervise someone in dept 5:

DEP5_EMPS   ← σDNO=5(EMPLOYEE)
RESULT1     ← πSSN(DEP5_EMPS)
RESULT2(SSN)← πSUPERSSN(DEP5_EMPS)
RESULT      ← RESULT1 ∪ RESULT2

Properties

  • UNION and INTERSECTION are commutative: R∪S = S∪R, R∩S = S∩R
  • Both are associative: R∪(S∪T) = (R∪S)∪T
  • DIFFERENCE is NOT commutative: R − S ≠ S − R (in general)
  • R ∩ S = R − (R − S) — intersection can be expressed using difference
2.13 CARTESIAN PRODUCT (×) 🔥 High-yield
📌 Definition

CARTESIAN (CROSS) PRODUCT combines tuples from two relations combinatorially — every tuple of R with every tuple of S.

R(A1,...,An) × S(B1,...,Bm) = Q(A1,...,An,B1,...,Bm)
  • Result has degree n + m attributes.
  • If |R| = nR tuples and |S| = nS tuples, then |R × S| = nR × nS tuples.
  • Operands do NOT need to be type-compatible.
⚠️ Common Mistake

Cartesian product alone is generally meaningless — it pairs up rows regardless of any real relationship between them. It only becomes useful once followed by a SELECT that filters to the truly related pairs (this is exactly how a JOIN is defined — see below).

💡 Example — CROSS PRODUCT + SELECT = meaningful query
FEMALE_EMPS   ← σSEX='F'(EMPLOYEE)
EMPNAMES      ← πFNAME,LNAME,SSN(FEMALE_EMPS)
EMP_DEPENDENTS← EMPNAMES × DEPENDENT
ACTUAL_DEPS   ← σSSN=ESSN(EMP_DEPENDENTS)
RESULT        ← πFNAME,LNAME,DEPENDENT_NAME(ACTUAL_DEPS)
2.14 JOIN Operations — Theta, Equijoin, Natural Join, Outer Joins 🔥🔥 Very Frequently Asked
📌 Definition

JOIN is a derivative of Cartesian Product — equivalent to performing a Cartesian product, then a SELECT using the join condition as the filter. It's one of the most difficult operations to implement efficiently in an RDBMS.

R ⋈<condition> S   ≡   σ<condition>(R × S)
Join TypeDefinition
Theta JoinGeneral join with any comparison operator (=, <, >, ≤, ≥, ≠) in the condition
EquijoinA theta join where the condition uses ONLY equality (=). Duplicate join-column appears in the result.
Natural Join (⋈)An equijoin over ALL common attribute(s) of the two relations, with the duplicate column removed automatically from the result
Left Outer JoinKeeps all tuples of the LEFT relation, padding unmatched right-side attributes with NULL
Right Outer JoinKeeps all tuples of the RIGHT relation, padding unmatched left-side attributes with NULL
Full Outer JoinKeeps all tuples from BOTH relations, padding with NULL wherever there is no match
SemijoinReturns tuples of R that have at least one matching tuple in S (only R's attributes are kept)
💡 Example — Equijoin vs Natural Join
-- Equijoin (duplicate clientNo column remains)
(πclientNo,fName,lName(Client)) ⋈ Client.clientNo=Viewing.clientNo
  (πclientNo,propertyNo,comment(Viewing))

-- Natural Join (duplicate column auto-removed)
(πclientNo,fName,lName(Client)) ⋈ (πclientNo,propertyNo,comment(Viewing))
Venn-diagram intuition for join types (shaded = kept in result)
⚠️ Common Mistake

Equijoin keeps both copies of the join attribute (e.g. two clientNo columns); Natural Join automatically drops the duplicate. This is the single most-tested distinction between the two.

QR.7 Common Exam Traps & Last-Minute Warnings 🔥🔥 Very Frequently Asked
⚠️ Trap 1

PROJECT (π) automatically removes duplicate rows — many students forget this and give the wrong tuple count.

⚠️ Trap 2

PROJECT is not commutative: π_A(π_B(R)) ≠ π_B(π_A(R)) in general — order of projected attributes changes the schema/output shape.

⚠️ Trap 3

Weak entity sets have no primary key of their own — they borrow the owner's key and add a partial/discriminator key.

⚠️ Trap 4

Data independence has two types — logical (conceptual schema changes don't affect external schema) and physical (internal schema changes don't affect conceptual schema). Don't mix them up.

⚠️ Trap 5

Total participation is drawn with a double line; partial participation with a single line — the opposite of what many students initially assume.
