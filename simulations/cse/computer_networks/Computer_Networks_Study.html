▶ Introduction to Computer Networks 🔥 High Yield

A computer network is a large number of separate but interconnected computers, forming a collection of autonomous computers connected by a single technology. Two computers are "interconnected" if they can exchange information, over wired or wireless links.

Definition The Internet is not a single network but a network of networks. The World Wide Web is a distributed system that runs on top of the Internet, presenting everything as a document (web page).

Uses of Computer Networks

  • Business applications: resource sharing, communication (email/messenger), e‑commerce with suppliers/customers, video conferencing.
  • Home applications: access to remote information, person-to-person communication, interactive entertainment, e-commerce, video-on-demand.
  • Mobile users: hotspots, texting, smartphones, e-books, GPS/m-commerce, wireless sensor networks, wearables.

Advantages vs Disadvantages

✔ Advantages
  • Enhances communication & availability of information
  • Convenient resource & file sharing
  • Highly flexible, inexpensive, cost-efficient
  • Boosts storage capacity
✘ Disadvantages
  • Lacks independence
  • Security difficulties, malware/virus spread
  • Requires efficient administration
  • Can lack robustness
Note A well-known example of a distributed system is the WWW — a middleware layer above the OS presents a single model/paradigm to users, hiding the underlying network of independent computers.

▶ Network Software: Protocols, Layers & Design Issues 🔥 High Yield
Key Definitions Protocol — an agreement between communicating parties on rules of communication (syntax, semantics, timing).
Peers — entities at the same layer on different machines.
Interface — primitive operations offered by a layer to the layer above it.
Network architecture — the set of layers + protocols.
Protocol stack — the list of protocols used by a system.

Protocol Hierarchies (Layering)

  • Networks are organized as a stack of layers, each built upon the one below it, to reduce design complexity.
  • Each layer offers services to the layer above, hiding implementation details — like a virtual machine.
  • Layer n on one machine "talks" to layer n on another — governed by the layer n protocol.
  • Violating the protocol makes communication difficult or impossible.

Service Primitives

Adjacent layers communicate via function calls called primitives (usually OS system calls). Common connection-oriented primitives:

PrimitiveMeaning
LISTENServer prepares for incoming connections (blocks)
CONNECTClient establishes connection with server (blocks)
ACCEPTServer unblocks + sends ACK, which unblocks client
RECEIVEServer accepts first client request (blocks)
SENDClient transmits request / server replies
DISCONNECTReleases the connection on both ends

Service vs Protocol 🔥 (common exam confusion)

Service
  • Set of primitives a layer provides to the layer above it
  • Defines what operations can be performed, not how
  • Relates to the interface between two layers (provider & user)
Protocol
  • Rules governing format/meaning of packets exchanged between peer entities
  • Entities use protocols to implement service definitions
  • Relates to packets between peers on different machines
Common Mistake Students often confuse "service" and "protocol" — remember: a service is a vertical relationship (between layers on the same machine); a protocol is a horizontal relationship (between peer layers on different machines).

Key Network Design Issues

  • Addressing — identifying senders/receivers
  • Rules for data transfer — one-way / two-way channels
  • Error control — detection & correction codes
  • Sequencing of messages — numbering
  • Flow control — buffers
  • Splitting long messages; MUX/DEMUX for multiple processes
  • Routing algorithms; Scalability; Security (confidentiality, integrity)
▶ OSI Reference Model (7 Layers) 🔥🔥 Very High Yield
Definition The OSI (Open Systems Interconnection) model is a proposal by ISO, standardizing protocols for connecting open systems. Revised in 1995. Its protocols are rarely used today, but the 7-layer model remains conceptually important.

Design Principles

  • A layer is created where a different abstraction is needed
  • Each layer performs a well-defined function
  • Layer functions chosen with international standardization in mind
  • Layer boundaries minimize information flow across interfaces
  • Number of layers should be optimum — not too many, not too few

The 7 Layers (top → bottom)

The OSI stack — data flows down through all 7 layers on the sender, and back up on the receiver.
#LayerMain Function
7ApplicationUser interface; protocols for file transfer, email, etc. (HTTP, FTP, DNS, SMTP)
6PresentationData format translation, encryption/decryption, compression/decompression
5SessionDialog control, token management, synchronization (checkpointing)
4TransportFirst true end-to-end layer; segmentation/reassembly, port addressing, reliability, flow & error control
3NetworkRouting, flow control, congestion control across heterogeneous networks
2Data LinkFrames bits, error detection/correction, flow control between adjacent nodes
1PhysicalTransmits raw bits; mechanical, electrical, timing interfaces
Mnemonic "All People Seem To Need Data Processing" (top to bottom: App, Pres, Session, Transport, Network, DataLink, Physical).

Layer Details Worth Remembering 🔥

  • Physical: volts per bit, bit duration, simplex/duplex, connector pins — all mechanical/electrical/timing aspects.
  • Data Link: breaks input into frames; processes acknowledgement frames; creates/recognizes frame boundaries using special bit patterns.
  • Network: overcomes heterogeneity — different addressing schemes, packet size limits, differing protocols between networks.
  • Transport: the first end-to-end layer — unlike the lower 3 layers which are "chained" hop-by-hop.
  • Session: manages "whose turn it is to transmit" (dialog control) + checkpointing for crash recovery.
  • Presentation: concerned with syntax/semantics of transmitted info — converts local representation to network standard.
▶ TCP/IP Reference Model 🔥 High Yield

Named after its two main protocols: TCP (Transmission Control Protocol) and IP (Internet Protocol). Designed in the 1970s; became the standard Internet protocol suite, connecting multiple networks seamlessly.

The 4 Layers

LayerFunction
Host-to-Network (Network Access)Not well-specified by the model; host connects to network via some protocol (varies by host/network) to send IP packets — e.g. twisted pair, fiber
InternetConnectionless layer defining protocol IP; delivers packets independently — may arrive out of order (higher layers handle ordering)
TransportTwo end-to-end protocols: TCP (connection-oriented, ordered, reliable, flow control) and UDP (connectionless, unreliable, used for real-time voice/video)
ApplicationNo session/presentation layers; contains higher-level protocols — TELNET, FTP, SMTP, HTTP, DNS
Example TCP guarantees ordered, error-free delivery — ideal for file downloads. UDP is connectionless and faster but unreliable — used for live video/voice calls where late data is worse than lost data.

Addressing at Different Layers 🔥

As data travels down the stack, each layer wraps the data with its own kind of address — this is why the same piece of data ends up carrying several different addresses by the time it hits the wire.

Port numbers identify a process on the host (Transport layer). IP addresses identify the host on the network (Network layer). MAC addresses identify the physical adapter on the local link (Data Link layer) — and get rewritten at every router hop.
Note This 3-tier addressing is exactly why a router changes the MAC address of a frame at every hop but never touches the IP address inside it — see the Router vs Bridge comparison in Module 2's Connecting Devices section.

▶ Physical Layer: Data Communication & Modes 🔥 High Yield
Definition Data communication is the exchange of data between two devices via a transmission medium. Effectiveness depends on 4 characteristics: Delivery, Accuracy, Timeliness, Jitter.

Components of a Data Communication System

5 components: Sender + Receiver (the devices), Message (the data), Transmission medium (the physical path), Protocol (the agreed-upon rules governing the exchange).
  1. Message — the information (text, numbers, images, audio, video)
  2. Sender — device sending the data
  3. Receiver — device receiving the data
  4. Transmission medium — the physical path (twisted-pair, coax, fiber, radio)
  5. Protocol — rules governing communication

Modes of Communication (Data Flow) 🔥

ModeDescriptionExample
SimplexUnidirectional — one device transmits, other only receives; uses full channel capacityKeyboard → Monitor
Half-DuplexBoth can transmit/receive, but not simultaneously; full capacity per direction, one at a timeWalkie-talkies
Full-DuplexBoth transmit & receive simultaneously; capacity divided or two physical paths usedTelephone network

Analog vs Digital

  • Analog data/signal — continuous, infinitely many intensity levels over time.
  • Digital data/signal — discrete states, limited defined values (commonly 1 and 0).
  • Signals can be periodic (repeats a pattern — a "cycle") or nonperiodic (no repeating pattern).

▶ Physical Topologies (Mesh, Star, Bus, Ring) 🔥🔥 Very High Yield

Physical topology = the geometric layout of links and nodes in a network. Four basic types: mesh, star, bus, ring (plus hybrid).

Gold nodes = devices. Mesh: every pair directly linked. Star: all links pass through a central hub (green). Bus: shared backbone cable. Ring: each node linked only to its two neighbors.

1. Mesh Topology

Every device has a dedicated point-to-point link to every other device.

Links needed (unidirectional) = n(n − 1)
Links needed (duplex mode)    = n(n − 1) / 2
Advantages
  • No traffic problems — dedicated links
  • Robust — one link failure doesn't cripple system
  • Privacy/security via dedicated lines
  • Easy fault identification & isolation
Disadvantages
  • Difficult installation/reconnection
  • Excessive cabling & expensive hardware (I/O ports)
  • Usually only used for backbones in hybrid networks

2. Star Topology

Each device has a dedicated point-to-point link only to a central hub; devices aren't directly linked to each other — the hub acts as an exchange.

Advantages
  • Less expensive than mesh
  • Easy install/reconfigure — one link/port per device
  • Robust — one link failure doesn't affect others
  • Easy fault isolation
Disadvantages
  • Single point of failure — the hub
  • More cabling than ring/bus

3. Bus Topology

One long backbone cable links all devices via drop lines and taps. Signal weakens (attenuates) with distance — limiting the number of taps and distance.

Advantages
  • Ease of installation
  • Uses less cabling than other topologies
Disadvantages
  • Difficult reconnection & fault isolation
  • Signal reflection at taps → quality degradation
  • Backbone fault stops all transmission

4. Ring Topology

Each device connects to only its two neighbors; signal passes in one direction, regenerated at each device's repeater.

Advantages
  • Easy install/reconfigure (neighbor-only links)
  • Simplified fault isolation
Disadvantages
  • Unidirectional traffic can be limiting
  • Single break can disable the entire ring

5. Hybrid Topology

Combines two or more topologies (bus + mesh + ring + star + tree), chosen based on business/institution needs, device count, location, and desired performance.

Common Mistake Don't confuse mesh's n(n-1)/2 link formula direction — this is for duplex links. For unidirectional (one-way per pair) links it's n(n-1), exactly double.

▶ Signal Encoding: Digital-to-Digital Conversion 🔥 High Yield

Signal encoding techniques fall into 4 categories: Digital data→Digital signal, Digital data→Analog signal, Analog data→Digital signal, Analog data→Analog signal. Focus here: Digital-to-Digital (line coding).

Data Element vs Signal Element

  • Data element — smallest unit of information = the bit (what we need to send).
  • Signal element — shortest unit of a digital signal (what we can actually send/carry).
  • r = ratio of data elements carried per signal element.

Data Rate vs Signal Rate

  • Data rate (N) — bits sent per second (bps) = bit rate.
  • Signal rate (S) — signal elements sent per second, measured in bauds = baud/modulation rate.
  • Goal: increase data rate while reducing baud rate.
Relationship between data rate (N) and signal rate (S):

        S = c × N × (1/r)

where c = case factor (worst/best/average), r = data elements per signal element
Average case typically uses c = 1/2
Solved Example r = 1 (one data element per signal element), bit rate N = 100 kbps, average c = 1/2.
S = (1/2) × 100,000 × (1/1) = 50,000 baud

Line Coding Schemes

Same 7-bit sequence shown across the four schemes — notice Manchester is the only one with a transition every single bit (self-clocking), while NRZ schemes can have long silent runs of no transitions.
SchemeDescription
Unipolar NRZAll signal levels on one side of axis; positive voltage = 1, zero voltage = 0; signal doesn't return to zero mid-bit
Polar NRZ-LTwo voltage levels; the level itself determines the bit value
Polar NRZ-IA change (or no change) in level determines the bit — transition = 1, no transition = 0
Manchester codingCombines NRZ-L + RZ; every symbol has a mid-bit transition (high→low or low→high); 2 voltage levels; used by IEEE 802.3 Ethernet 🔥
Differential ManchesterCombines NRZ-I + RZ; every symbol has a mid-bit transition, but the starting level depends on the bit value
Note Manchester encoding is used by classic Ethernet — a frequently tested fact. Remember: a binary 1 = high-then-low; binary 0 = low-then-high (within each bit period).
▶ Performance Indicators: Bandwidth, Throughput, Latency, BDP, Jitter 🔥🔥 Very High Yield — Numericals!

Performance metrics measure a network's Quality of Service (QoS) as perceived by the user: Bandwidth, Throughput, Latency (Delay), Bandwidth-Delay Product, Jitter.

1. Bandwidth

  • Bandwidth in Hertz — range of frequencies in a composite signal/channel: Bandwidth = upper frequency − lower frequency
  • Bandwidth in bits per second — the number of bits/sec a channel can transmit (a "speed" measure)
  • "Bandwidth" = Capacity; "Speed" = Transfer rate. Data rate is directly proportional to bandwidth (Hz).
Solved Example — Bandwidth A composite signal has frequencies 100–5000 Hz (5 sine waves: 100,300,500,700,900 Hz).
Bandwidth B = f_high − f_low = 900 − 100 = 800 Hz

2. Throughput

  • Number of bits successfully transmitted per unit time — how fast data can actually be sent.
  • Bandwidth = potential measurement; Throughput = actual measurement (always ≤ bandwidth, limited by SNR & hardware).
Throughput = (frames × bits per frame) / time

Example: 10 Mbps link, 12,000 frames/min, 10,000 bits/frame
Throughput = (12,000 × 10,000) / 60 = 2,000,000 bps = 2 Mbps
(≈ one-fifth of the 10 Mbps bandwidth)

3. Latency (Delay) 🔥

Time for an entire message to completely arrive at the destination — from first bit sent to last bit delivered.

Latency = Propagation Time + Transmission Time + Queuing Time + Processing Delay

Propagation Time  = Distance / Propagation Speed
Transmission Time = Message size / Bandwidth
  • Propagation delay — time for one bit to travel from source to destination.
  • Transmission delay — time for the entire message to be pushed onto the link.
  • Queuing time — time a frame waits at an intermediate device (router); increases with network load.
  • Processing delay — time for a router to determine where to forward a packet.
  • Low-Latency networks = small delays; High-Latency networks = long delays → causes bottlenecks.
Solved Example — Propagation Time Distance = 12,000 km, propagation speed = 2.4×10⁸ m/s.
Propagation time = (12,000 × 1000) / (2.4×10⁸) = 50 ms
Solved Example — Short Message, High Bandwidth 2.5 kbyte message, bandwidth = 1 Gbps, distance = 12,000 km, speed = 2.4×10⁸ m/s.
Propagation time dominates; transmission time is negligible and can be ignored.
Solved Example — Long Message, Low Bandwidth 5 Mbyte message (image), bandwidth = 1 Mbps, distance = 12,000 km.
Transmission time dominates; propagation time is negligible and can be ignored.

4. Bandwidth–Delay Product 🔥

Definition The product of a link's bandwidth and its delay gives the maximum number of bits that can "fill" the link at any instant (i.e., the number of bits in transit). 
Bandwidth-Delay Product = Bandwidth × Delay

Case 1: Bandwidth = 1 bps, Delay = 5s
BDP = 1 × 5 = 5 bits can fill the link at any given time

Case 2: Bandwidth = 5 bps, Delay = 5s (same delay, wider pipe)
BDP = 5 × 5 = 25 bits can fill the link at any given time
Same delay (5s), but a wider pipe (Case 2: 5 bps vs Case 1: 1 bps) holds 5× more bits in flight — 25 bits instead of 5. Bandwidth is literally how "fat" the pipe is; delay is how "long" it is.

5. Jitter

  • Jitter = "packet delay variance" — occurs when different packets face different delays and the receiving application is time-sensitive (audio/video).
  • Measured in milliseconds (ms).
  • Can cause flickering in video displays.
  • Jitter buffer mitigates jitter by buffering packets before feeding them out at a steady rate (e.g. YouTube buffering before playback).
Example If the delay for the first packet is 10 ms, for the second is 35 ms, and for the third is 50 ms, the real-time destination application experiences jitter — the uneven gaps between packet arrivals (10ms, then 25ms, then 15ms) disrupt smooth playback even though every packet eventually arrives.
Common Mistake Don't confuse Bandwidth (potential/capacity) with Throughput (actual achieved rate) — throughput is always less than or equal to bandwidth in practice.

▶ Data Link Layer: Design Issues & Services 🔥 High Yield

The DLL's data unit is the frame (not individual bits, as in the physical layer). DLL design deals with algorithms for reliable, efficient communication between two adjacent machines.

Problems the DLL Must Solve

  • Communication channels make errors occasionally
  • Channels have finite data rate + nonzero propagation delay
  • These limitations affect data transfer efficiency

Functions of the Data Link Layer

  1. Well-defined service interface to the network layer
  2. Frame management (encapsulating packets into frames: header + payload + trailer)
  3. Dealing with transmission errors (error detection & correction)
  4. Flow control — preventing fast senders from swamping slow receivers

Services Provided to the Network Layer 🔥

ServiceDescriptionBest Used For
Unacknowledged connectionlessIndependent frames sent, no ack, no logical connection, no loss recoveryLow error-rate channels; real-time traffic (voice); most LANs
Acknowledged connectionlessNo logical connection, but each frame individually acknowledged & resent if neededUnreliable channels like wireless
Acknowledged connection-orientedConnection established first; each frame numbered; guaranteed exactly-once, in-order deliveryLong unreliable links (satellite, long-distance telephone)

The connection-oriented service has 3 phases: (1) Connection establishment (initialize counters/variables), (2) Data transfer, (3) Connection release (free buffers/resources).

Example A VoIP call over Wi-Fi uses unacknowledged connectionless service — a dropped voice frame just causes a tiny glitch, not worth the delay of retransmission. A file transfer over a shaky satellite link uses acknowledged connection-oriented service — every single frame must arrive, in order, exactly once.
▶ Framing Methods 🔥🔥 Very High Yield

Breaking a bit stream into frames is harder than it looks — the receiver must easily find frame boundaries while using minimal bandwidth. Four methods exist:

1. Byte/Character Count

  • A header field specifies the number of bytes/characters in the frame.
  • Disadvantage: if the count is garbled by a transmission error, the destination loses synchronization — and retransmission requests don't help since the destination doesn't know how many bytes to skip. Rarely used.

2. Flag Bytes with Byte (Character) Stuffing

  • Each frame starts/ends with a special FLAG byte; two consecutive flags = end of one frame + start of next.
  • Problem: the flag pattern might occur naturally in the data (especially binary data).
  • Solution: sender inserts an escape byte (ESC) before every accidental flag byte in the data — called byte stuffing. Receiver removes the ESC bytes before passing data up.
  • If an ESC byte occurs naturally in data, it too is stuffed with another ESC (doubled ESC = natural occurrence).
  • Used in PPP (Point-to-Point Protocol).
  • Disadvantage: tied to 8-bit characters — not all character codes use 8 bits.
Example — Byte Stuffing If the data itself contains a FLAG byte, the sender inserts an ESC just before it: ...DATA ESC FLAG DATA.... The receiver sees the ESC, discards it, and treats the following byte as ordinary data instead of a real delimiter. If the data contains an ESC byte, it gets doubled: ...DATA ESC ESC DATA....

3. Flag Bits with Bit Stuffing 🔥

  • Bit-level framing — frames can hold an arbitrary number of bits. Developed for HDLC.
  • Each frame begins/ends with the bit pattern 01111110 (0x7E) — the FLAG byte.
  • Whenever the sender's DLL sees five consecutive 1s in the data, it automatically stuffs a 0 bit after them.
  • Receiver, upon seeing 5 consecutive 1s followed by a 0, automatically destuffs (removes) that 0 — fully transparent to the network layer.
  • Also ensures a minimum density of transitions for synchronization — reason USB uses bit stuffing.
Example — Bit Stuffing Data: 01001111110111110
After adding flags: 01111110 01001111110111110 01111110
After bit stuffing: 01111110 0100111110101111100 01111110

4. Physical Layer Coding Violations

  • Used only where the encoding method includes redundancy (e.g. some LANs encode 1 bit using 2 physical levels: high-low = 1, low-high = 0).
  • Combinations high-high / low-low are unused for data → reserved to delimit frames.
  • Advantage: easy to find frame boundaries, no need to stuff data.

Byte Stuffing vs Bit Stuffing 🔥

Byte StuffingBit Stuffing
Larger increase in frame length — worst case roughly doubles frame size if data is all flag-pattern bytesSmaller increase — roughly 12.5% overhead (1 bit added per 5 consecutive 1s)
Note Most real protocols use a combination of byte count + one other method for extra safety. If the delimiter/checksum doesn't match at the expected position, the stream is re-scanned for the next delimiter.
▶ Error Detection & Correction: Hamming, Parity, Checksum, CRC 🔥🔥 Very High Yield — Numericals!

Types of Errors

  • Single-bit error — only 1 bit changes.
  • Burst error — 2 or more bits change; harder to correct, but since data is sent in blocks, low error rates mean only 1–2 blocks may be affected.

Detection vs Correction

Error Detection
  • Simple yes/no — just checks if an error occurred
  • Not concerned with number/location of errors
  • Triggers retransmission
  • Best on reliable channels (fiber) — cheaper to just retransmit
Error Correction (FEC)
  • Needs to know exact count & location of corrupted bits
  • Also called Forward Error Correction
  • Needs more redundant bits than detection
  • Best on unreliable channels (wireless) — retransmission itself may fail

4 error-correcting code families: Hamming codes, Binary convolutional codes, Reed-Solomon codes, Low-Density Parity Check (LDPC) codes.

Frame = k data bits + r redundant (check) bits;  n = k + r  (n-bit "codeword")

Constraint on r:     2^r ≥ k + r + 1
Example: if k = 7, the smallest valid r = 4

Hamming Distance 🔥

Definition The Hamming distance between two codewords = the number of bit positions in which they differ. Found via XOR of the two codewords, then counting the 1s.
Solved Examples d(000, 011) = 2  |  d(10101, 11110) = 3

The Hamming distance of a complete code = the minimum HD between all possible pairs of legal codewords. With r check bits, only 1/2ʳ of all possible messages are legal codewords — this "sparseness" is what enables error detection/correction.

Significance of Hamming Distance (dmin) 🔥🔥

To reliably DETECT  d errors → need a code with distance ≥ d + 1
To reliably CORRECT d errors → need a code with distance ≥ 2d + 1
Solved Example A code has dmin = 4.
Detection: can detect 3-bit errors (3+1=4)
Correction: can correct only 1-bit errors (since 1.5 isn't valid — needs odd dmin)
Rule: error-correcting codes need an odd minimum distance (3, 5, 7…)
Solved Example — Where Correction Breaks Down A code has only 4 valid codewords: 0000000000, 0000011111, 1111100000, 1111111111. Its dmin = 5, so it can detect 4-bit errors (4+1=5) and correct 2-bit errors (2×2+1=5).

Correction works: if 0000000111 arrives, the receiver correctly concludes the original was 0000011111 (closest codeword, only 2 bits off).

Correction fails: if a triple-bit error changes 0000000000 into 0000000111, the receiver makes the exact same guess — 0000011111 — which is now wrong. A 3-bit error exceeds this code's 2-bit correction capability (2×3+1=7 > 5), so it gets "corrected" to the wrong codeword instead of being flagged.

Hamming Code — Single Bit Error Correction 🔥🔥

  1. Number codeword bits starting at 1. Positions that are powers of 2 (1, 2, 4, 8, 16…) are parity bits; all others hold data.
  2. Each parity bit at position p checks bit positions whose binary representation has that power-of-2 bit set (e.g., bit 11 = 1+2+8, so checked by parity bits 1, 2, 8).
  3. Set each parity bit per the even/odd parity rule.
  4. At the receiver: recheck each parity bit; sum the positions of all incorrect parity checks — this sum directly gives the position of the bad bit (the "error syndrome"). Flip that bit to correct it.
Positions 1, 2, 4, 8 (powers of 2) hold parity bits; all other positions hold data bits — codeword shown is the worked example below (011100101010).
Worked Example — 8 data bits: 10011010 Codeword positions: P1 P2 D1 P4 D2 D3 D4 P8 D5 D6 D7 D8 → skeleton _ _ 1 _ 0 0 1 _ 1 0 1 0
Computed (even parity): Codeword = 011100101010
If received as 011100101110: positions 2 & 8 fail parity → 2+8=10 → bit 10 is flipped → recovers 011100101010 ✓
Worked Example 2 — Receiver Side: 8 data bits 11000111 Sender computes and transmits codeword 111010010111. In transit, the bit at position 5 flips.
Receiver recomputes all 4 parity checks on the corrupted codeword 111000010111:
Parity bitChecks positionsResult
P11,3,5,7,9,11fails (contributes 1)
P22,3,6,7,10,11passes (contributes 0)
P44,5,6,7,12fails (contributes 4)
P88,9,10,11,12passes (contributes 0)
Syndrome Add up the positions of every failed check: 1 + 4 = 5. That sum is the exact position of the corrupted bit — flip bit 5 and the original codeword 111010010111 is recovered. This "add up the failed parity positions" trick is the fast way to solve Hamming problems in an exam without redrawing the whole 12-bit table each time.

Error-Detecting Codes: Parity, Checksum, CRC

All three are linear, systematic block codes — cheaper than correction, ideal for low-error channels (copper/fiber).

MethodHow it worksNotes
Parity BitOne bit appended so total 1-bits is even (or odd) — equivalent to XOR of data bitsDistance = 2, detects only single-bit errors reliably. For burst protection, arrange data as a matrix and use row/column parity (interleaving) — detects up to k bit errors per row/column.
ChecksumRunning sum of data "words", then complement taken; appended at message end. Receiver sums everything (incl. checksum) — result of 0 means no error.e.g., 16-bit Internet checksum used on all IP packets.
CRC (Cyclic Redundancy Check)Treats bit strings as coefficients of a polynomial; sender & receiver agree on generator polynomial G(x). Frame is made divisible by G(x) via modulo-2 division.Also called polynomial code; strongest of the three. Detects all single-bit errors and all burst errors of length ≤ r (the degree of G(x)).
 
CRC Algorithm:
1. Let r = degree of G(x). Append r zero bits to the frame → represents x^r · M(x)
2. Divide this bit string by G(x) using modulo-2 (XOR) division
3. Subtract the remainder from x^r · M(x) (mod-2) → gives the transmitted codeword T(x)
4. Receiver divides received T(x) by G(x): remainder 0 = no error, nonzero = error detected
Common Mistake Polynomial arithmetic in CRC is modulo 2 — there are NO carries in addition and NO borrows in subtraction; both operations are simply XOR.
Worked Example — CRC Division Data = 1101011011, Generator G(x) = 10011 (degree r = 4).
1. Append 4 zero bits: 11010110110000
2. Modulo-2 divide by 10011 → remainder = 1110
3. Transmitted codeword = data + remainder = 11010110111110
4. Receiver divides the codeword by 10011 again — remainder 0000 confirms no error.
Example — Internet Checksum Sender splits data into 16-bit words, adds them with end-around carry, then sends the 1's complement of the sum as the checksum. Receiver adds all words (including the checksum) — a result of all 1s (equivalently, complement = 0) means the data arrived intact.
Worked Example — Checksum (simplified, 8-bit words) Three data bytes: 11001100, 10101010, 01110111.
1. Add them together (with end-around carry — any overflow bit wraps back and gets added in): sum = 11101110
2. Checksum = 1's complement of the sum = 00010001 — this is what gets sent alongside the data.
3. Receiver adds all 3 data bytes and the checksum together: result = 11111111 (all 1s) → no error detected. Any single corrupted bit would break this all-1s result.
Note — Overhead Comparison For a 1000-bit block with 10⁻⁶ error rate: Error correction needs ~10,000 check bits/megabit (10 check bits per 1000-bit block). Error detection + retransmission needs only ~2001 bits/megabit (1 parity bit/block + occasional retransmission) — far more efficient when errors are rare!
▶ Flow Control & Sliding Window Protocols 🔥🔥 Very High Yield — Numericals!

Flow Control vs Error Control

  • Flow control restricts how much data a sender can send before waiting for acknowledgement.
  • Error control uses ARQ (Automatic Repeat reQuest) = acknowledgements + timeouts to guarantee reliable delivery.

ARQ Basics

  • ACK — small control frame (header only, no data) confirming receipt.
  • Timeout — if no ACK arrives in a reasonable time, the sender retransmits.

Stop-and-Wait Protocol 🔥

  • Sender sends one frame, then waits for an ACK before sending the next.
  • Uses a 1-bit sequence number (0 or 1) to detect duplicate frames (in case an ACK is lost/delayed).
Stop-and-Wait ARQ = Stop-and-Wait + Timer/Timeouts + Sequence Numbers
Common Mistake Without sequence numbers, a retransmitted frame after a lost ACK looks identical to the "next" frame to the receiver — causing duplicate delivery to the network layer. Sequence numbers fix this.
Disadvantage Stop-and-Wait is bandwidth-inefficient — the sender is idle after every frame, waiting for an ACK. Piggybacking (combining an ACK with an outgoing data frame in bidirectional links) helps, but the real fix is Sliding Window.

Sliding Window Protocols — Core Idea 🔥

  • Applies pipelining — sender can transmit multiple frames before needing an ACK.
  • Sender's window = set of sequence numbers it's permitted to send (unacknowledged/outstanding frames).
  • Receiver's window = set of sequence numbers it's willing to accept.
  • Sequence numbers use m header bits → range 0 to 2ᵐ−1.
  • Window slides forward as ACKs arrive (lower edge) and new frames are sent (upper edge).

Go-Back-N ARQ

F2 is lost in transit. The receiver silently discards the out-of-order frames F3 and F4 that follow. When the sender's timer for F2 expires, it goes back and resends F2, F3, F4 — even though F3 and F4 were sent correctly the first time.
  • Sender window size = 2^m − 1; Receiver window size = 1.
  • One timer for the oldest outstanding (unacknowledged) frame.
  • Receiver sends a positive ACK only for frames arriving safe & in order; if a frame is damaged/out-of-order, receiver stays silent and discards all subsequent frames (no buffering of out-of-order frames).
  • On timeout, sender resends all frames starting from the one with the expired timer — hence "Go Back N".
  • Supports cumulative ACK — ACK2 implicitly acknowledges ACK1 too.

Selective Repeat ARQ

Unlike Go-Back-N, the receiver keeps F2 and F3 in a buffer instead of discarding them, and tells the sender exactly which frame (F1) is missing via a NAK — so only that one frame needs to be resent.
  • Window size (sender = receiver) reduced to 2^m / 2.
  • Uses NAK (negative ACK) reporting the sequence number of a damaged frame, before the timer expires.
  • Receiver buffers correctly-received out-of-order frames; only resends the specific missing/damaged frame.
  • More efficient over noisy links, but needs more receiver memory/complexity.
  • Delivery to network layer requires: (1) a set of consecutive frames has arrived, (2) that set starts from the window's beginning.

▶ HDLC (High-Level Data Link Control) Protocol 🔥 High Yield

HDLC is a bit-oriented, synchronous data link layer protocol by ISO, used over point-to-point and multipoint links. Implements Stop-and-Wait; supports error & flow control.

Station Types & Transfer Modes

  • Station types: Primary, Secondary, Combined.
  • Normal Response Mode (NRM): unbalanced configuration — 1 primary + multiple secondary stations. Primary sends commands; secondaries only respond. Used point-to-point & multipoint.
  • Asynchronous Balanced Mode (ABM): point-to-point only; each station acts as both primary & secondary (peers). The commonly used mode.

Frame Types

  • I-frames (Information) — carry user data + control info (piggybacking)
  • S-frames (Supervisory) — carry control information only
  • U-frames (Unnumbered) — reserved for link/system management

Frame Format

FieldSizeUse

Address1+ byte(s)To/From address depending on primary/secondary; last bit of last byte = 1
Control1–2 bytesFrame & error control; interpretation differs by frame type
InformationvariableUser data from network layer
FCS2 or 4 bytesFrame Check Sequence — error detection

Control Field Formats

  • I-frame: bit 0 = 0 (type flag); next 3 bits = N(S) sequence number; last 3 bits = N(R) piggyback ack number; includes P/F (Poll/Final) bit.
  • S-frame: first 2 bits = 10; last 3 bits = N(R); bits 3–4 = code identifying S-frame type:
CodeTypePurpose
00RR (Receive Ready)Acknowledges safe receipt
10RNR (Receive Not Ready)ACK + "I'm busy" (congestion control)
01REJ (Reject)NAK used in Go-Back-N ARQ
11SREJ (Selective Reject)NAK used in Selective Repeat ARQ
  • U-frame: first 2 bits = 11; used for session management (32 different types possible).
Example — Piggybacking in Action Station A sends I-frame with N(S)=2 (its 3rd frame) to B. B has data to send back too, so instead of a bare ACK, B replies with an I-frame carrying N(S)=0 (its own data) and N(R)=3 (acknowledging it has correctly received A's frames up through sequence 2) — one frame does double duty as data + ACK.
▶ MAC Sublayer: Channel Allocation, ALOHA, CSMA/CD 🔥🔥 Very High Yield

DLL splits into LLC (Logical Link Control — one protocol for all IEEE LANs) and MAC (Medium Access Control — different per LAN type). MAC solves: who gets to use a shared/broadcast channel?

Channel Allocation: Static vs Dynamic

Static AllocationDynamic Allocation
FDM (Frequency Division Multiplexing) — bandwidth split into N fixed bands; used in FM radio, GSMAllocation adapts to current demand; called Multiple Access Protocols

ALOHA 🔥

Developed by Norman Abramson (1970) for radio LANs; applicable to any shared-channel system.

Pure ALOHA's free-for-all timing doubles the danger window compared to Slotted ALOHA's synchronized slots — this is exactly why vulnerable time is 2t vs t.
Pure ALOHASlotted ALOHA
Transmit whenever data is ready — continuous timeTime divided into discrete slots; must wait for slot boundary — discrete time
No global time synchronization requiredRequires synchronization (clocks)
Higher collision probabilityImproved efficiency over pure ALOHA
Vulnerable time = 2t (t = frame transmission time)Vulnerable time = t (frame transmission time only)

On collision, sender waits a random time before retrying (a "contention system") — different wait times reduce repeated collision probability.

Example — Vulnerable Time A LAN sends 1000-bit frames at 1 Mbps → frame transmission time t = 1000/1,000,000 = 1 ms.
Pure ALOHA vulnerable time = 2t = 2 ms. Slotted ALOHA vulnerable time = t = 1 ms — half the risk, because transmissions can only begin at slot boundaries.

Carrier Sense Multiple Access (CSMA)

Improves on ALOHA — station senses the medium before transmitting. Reduces (but can't eliminate) collisions due to propagation delay.

VariantBehavior
1-Persistent CSMAIf channel busy, keep sensing continuously; transmit with probability 1 the instant it's idle. Higher collision risk if 2 stations were both waiting.
Non-Persistent CSMAIf busy, don't keep sensing — wait a random time, then re-check. Better utilization, longer delays than 1-persistent.
p-Persistent CSMAIf idle, transmit with probability p; defer to next slot with probability q=1−p. Repeats until sent or channel busy.

CSMA/CD (Collision Detection) 🔥🔥

Both stations sense the channel idle and transmit almost simultaneously → signals overlap (collision) → both send a jam signal and back off for a random time before retrying.
  • Station continues listening while transmitting — aborts immediately on detecting a collision, saving time/bandwidth.
  • Basis of classical Ethernet. Uses signal power/pulse-width to detect collisions. No ACK used.
  • Steps: (1) Check ready → (2) sense channel idle → (3) transmit + monitor for collision; if collision, stop, send jam signal, wait random time, retry → (4) if no collision, complete transmission.
  • CSMA/CD can be in one of 3 states: contention, transmission, idle.

Collision-Free Protocols

ProtocolHow it works
Bit-Map ProtocolEach contention period has N slots; station j announces intent by placing a 1 bit in slot j. After N slots, all stations know who wants to transmit — they go in numerical order (a "reservation protocol")
Binary CountdownStations broadcast their binary address bit-by-bit (high-order first); bits are OR-ed on the channel. If a station sees its 0-bit overwritten with 1, it withdraws. Scales better than bit-map (avoids 1-bit-per-station overhead)
Token PassingA "token" message circulates in a fixed order; only the token holder may transmit, then passes the token on
Example — Binary Countdown Stations with addresses 1001, 1010, and 0111 all want to transmit. They broadcast their address bit by bit, high-order first, and the channel OR-combines the bits: after bit 1, the combined value is 1 (since 1001 and 1010 both send 1) — station 0111 sees its 0 overwritten by a 1 and immediately withdraws. After bit 2, both remaining stations send 0, so both continue. After bit 3, 1001 sends 0 and 1010 sends 1 — 1001 sees its bit overwritten and withdraws. Station 1010 wins (highest address) and transmits first.
Common Mistake CSMA/CD ≠ CSMA/CA. CSMA/CD (Collision Detection) is for wired Ethernet; CSMA/CA (Collision Avoidance, via RTS/CTS like MACA) is for wireless — because wireless stations often can't detect their own collisions.

Ethernet Addressing (MAC Address)

  • 6-byte address, burnt into ROM on the adaptor, written as 6 hex-pairs separated by colons (e.g. 8:0:2b:e4:b1:2).
  • Manufacturer prefix allocated per vendor (e.g. AMD = 8:0:20).
  • Unicast — 2nd hex digit from left is even.
  • Multicast — 2nd hex digit from left is odd.
  • Broadcast — all digits are F (FF:FF:FF:FF:FF:FF).
  • MAC addresses can be altered — known as MAC spoofing.
Example — Reading a MAC Address Address 01:00:5E:00:00:01: the first byte is 01 = binary 00000001 — the low-order bit of the first byte is 1, marking it as a multicast address (this specific one is IPv4 multicast). Address FF:FF:FF:FF:FF:FF is the reserved broadcast address — every station on the LAN accepts it.

Transmitter Algorithm & Collisions 🔥

  • If line idle → transmit immediately. If busy → wait until idle, then transmit.
  • On collision detection: send a 32-bit jamming sequence, then stop. Minimum sent during collision = 96 bits (64-bit preamble + 32-bit jam).
  • Worst-case round-trip propagation time on 10Mbps Ethernet = 51.2 μs (defines the collision slot time).
  • To guarantee detecting a collision on its own frame, a transmitter needs to send ≥512 bits → hence minimum Ethernet frame = 512 bits = 64 bytes (14 header + 46 data + 4 CRC).
Minimum frame size derivation:
Delay × Bandwidth = 51.2 × 10⁻⁶ s × 10 × 10⁶ bps = 512 bits = 64 bytes

Binary Exponential Backoff Algorithm 🔥🔥

After collision number n, wait k × 51.2 μs,
where k is randomly chosen from 0 to (2ⁿ − 1)

Collision 1: choose k from {0,1}         → range 2^1
Collision 2: choose k from {0,1,2,3}     → range 2^2
Collision 3: choose k from {0..7}        → range 2^3
...
After 10 collisions: randomization interval frozen at 1023 slots (2^10)
After 16 collisions: controller reports failure to computer
Worked Example — Backoff Delay After the 3rd collision, a station picks k randomly from {0,1,2,...,7} (range 2³). If it picks k=5, it waits 5 × 51.2 μs = 256 μs before its next attempt. Two stations colliding repeatedly become increasingly unlikely to pick the same k as the range widens — this is what makes the algorithm converge.

Switched Ethernet

  • A switch has a high-speed backplane with 4–32 plug-in line cards, 1–8 connectors each.
  • Each connector = 10Base-T twisted-pair connection to a single host.
  • Frame is copied locally if destined for the same card; otherwise sent over the backplane (many Gbps, proprietary protocol) to the destination card.
▶ Wireless LAN & IEEE 802.11 🔥 High Yield

Why Wireless is Different

  • In wired LAN, when one station sends, all others receive it. In wireless, limited radio range means a station may not reach every other station.
  • Radio signals reflect off objects (multipath fading); mobility requires reconfiguration when moving between base stations.

Hidden Terminal & Exposed Terminal Problems 🔥🔥

Hidden Terminal Problem

A talks to C. B can't hear A (out of range) and thinks C is idle, so B also transmits to C — causing a collision at C. B is "hidden" from A.

Exposed Terminal Problem

A is talking to some other station. C is in range of A and senses the channel busy, so C refrains from talking to B — even though B and C could safely communicate. A is "exposed" to C.

MACA (Multiple Access with Collision Avoidance)

  • Sender stimulates the receiver into sending a short frame first, alerting nearby stations to stay silent.
  • RTS (Request To Send, ~30 bytes, contains data length) sent by A → CTS (Clear To Send, echoes data length) sent back by B → A then transmits data.
  • Stations hearing RTS stay silent (near A); stations hearing CTS stay silent (near B) — solves the hidden terminal problem only.
  • Collisions still possible if RTS frames from 2 senders collide — resolved via random backoff.

MACAW (MACA for Wireless)

  • Improves MACA: adds ACK frames at DLL for each successful data frame.
  • Adds carrier sensing (avoids simultaneous RTS).
  • Runs backoff separately per source-destination stream.
  • Solves both hidden and exposed terminal problems.
Example In plain MACA, C (exposed to A but wanting to talk to D) would still stay silent after overhearing A's RTS — even though a C→D exchange wouldn't disturb A/B's link. MACAW's added carrier sensing and ACK-based bookkeeping let C recognize this and transmit safely, recovering wasted capacity.

IEEE 802.11 (Wi-Fi) Basics

  • Operates on 2.4 GHz, 5 GHz, 6 GHz, and 60 GHz frequency bands.
  • BSS (Basic Service Set) — building block: wireless stations + optional central AP (Access Point).
  • ESS (Extended Service Set) — 2+ BSSs connected via a distribution system (usually a wired LAN).
  • Two configurations: infrastructure mode (with AP/base station) and ad-hoc mode (without one).

802.11 MAC Modes

802.11 Frame Structure

  • Frame Control fields: Version, Type (Data/Control/Management), Subtype (RTS/CTS), To-DS, From-DS, More Fragments, Retry, Power, More Data, WEP encryption flag, Strict Order flag.
  • Duration — channel occupation time; Four Address fields (source, dest, + 2 base station addresses); Sequence (16 bits: 12 for frame ID, 4 for fragment); Data (up to 2312 bytes); Checksum (4-byte CRC).
  • Frame types: Management (single-cell only), Control (RTS/CTS/ACK — short, few addresses), Data (actual payload).
Note — Why 4 Addresses? A wired Ethernet frame only ever needs source + destination. In infrastructure-mode Wi-Fi, a frame often bounces through an Access Point, so the MAC layer needs to separately track "who physically sent this hop" and "who is the original/final endpoint." The 4th address only comes into play for AP-to-AP wireless distribution links.
▶ Connecting Devices: Repeaters, Hubs, Bridges, Switches, Routers, Gateways 🔥 High Yield

Devices used to connect LANs/WANs together or to the Internet, operating at different OSI layers.

Layers of Operation

Passive HubBelow Physical Layer

Router / Three-layer SwitchPhysical + Data Link + Network (Layers 1–3)

Device Details

  • Passive Hub — just a wire connector; in star topology it's the collision point.
  • Repeater — regenerates a weakening/corrupted signal, physical layer device; used to overcome bus-topology length limits.
  • Active Hub — a multiport repeater; when A sends to B, hub forwards to all ports; only B keeps the data, others discard it. Has no data-link address awareness.
    • A hub is "dumb" — it repeats to everyone. A bridge is address-aware — it looks up the destination MAC and only forwards out the port that actually leads there.
  • Bridge — regenerates signal (physical layer) AND checks MAC addresses (data link layer) for filtering. Maintains a table for forwarding decisions.
    • Transparent bridge — stations are unaware it exists; must (1) forward frames, (2) auto-learn its table, (3) prevent loops.
    • Learning bridge — dynamically builds its address→port table by inspecting source addresses of incoming frames (destination address used for the forwarding decision itself).
  • Switch — an intelligent multiport bridge; uses MAC addresses + packet switching; supports unicast/multicast/broadcast; full-duplex; supports error checking.
  • Router — a 3-layer device; regenerates signal, checks physical (MAC) address, AND checks network-layer (IP) address. Connects LANs, WANs, or LANs+WANs into an internetwork.
  • Gateway — operates across all 5 (TCP/IP) or 7 (OSI) layers; reads and interprets application messages; connects two internetworks using different models; can provide security by filtering unwanted application-layer messages.

AspectRepeater / BridgeRouter

Common Mistake A frequent exam trap: students say bridges and routers both "forward based on address" without distinguishing that a bridge forwards using the unchanged MAC address, while a router rewrites the MAC address at each hop (based on IP routing) while keeping the IP address intact end-to-end.
Example — Bridge vs Router Forwarding Host X sends a frame to Host Y across 3 hops. A bridge at each hop just checks the destination MAC address and forwards the exact same frame unchanged — X's and Y's MAC addresses stay in the frame the whole way. A router at each hop strips the old MAC header, looks at the destination IP address to decide the next hop, and builds a brand-new MAC header (its own outgoing interface MAC as source, the next router/host's MAC as destination) — while the source/destination IP addresses never change end-to-end.
